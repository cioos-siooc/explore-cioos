# db-loader

Tool to upload harvested data to the CIOOS Explorer database

## Overview

The db-loader takes CSV files generated by the harvester and loads them into the PostgreSQL database. It supports two modes:
- **Full reload mode** (default): Deletes all existing data and reloads everything from scratch
- **Incremental mode** (new): Updates only the datasets in the CSV files, preserving other data

## Installation

```bash
cd db-loader
python -m venv venv
source ./venv/bin/activate
pip install .
```

## Usage

### Full Reload Mode (Default)

This is the original behavior - it clears all data and reloads everything:

```bash
python -m cde_db_loader --folder harvest
```

This mode:
1. Drops all database constraints
2. Deletes ALL data from all tables (datasets, profiles, organizations, etc.)
3. Inserts all datasets and profiles from CSV files
4. Rebuilds hexes and lookup tables
5. Recreates constraints

**Use when**: You want a complete refresh of the database from all ERDDAP servers

### Incremental Mode (Recommended for Updates)

Use the `--incremental` flag to update only specific datasets:

```bash
python -m cde_db_loader --folder harvest --incremental
```

This mode uses an optimized workflow with temporary tables and SQL functions:

1. **Creates temporary staging tables** without constraints (via `create_temp_tables()`)
2. **Loads CSV data** into temp tables (no constraint validation)
3. **Processes data** in temp tables (calculates geometry, links datasets, etc.)
4. **UPSERT datasets**: Updates existing or inserts new ones using `(dataset_id, erddap_url)` as unique key
5. **Replaces profiles**: Deletes old profiles for updated datasets, then inserts new ones
6. **UPSERT skipped_datasets**: Updates the list of skipped datasets
7. **Runs processing functions**: Updates CKAN data, recalculates hexes, rebuilds indexes
8. **Preserves other data**: Datasets not in the CSV files remain unchanged

**Use when**:
- You harvested only specific datasets (using `dataset_ids` filter in harvest config)
- You want to update a subset of datasets without affecting others
- You want faster database updates (typically 5-10x faster than full reload)
- You're running scheduled/automated updates

## Workflow Examples

### Example 1: Update Specific Datasets

```bash
# 1. Edit harvest_config.yaml to specify datasets
cat > harvest_config.yaml <<EOF
erddap_urls:
  - https://cioos-atlantic-ckan.cioosatlantic.ca/erddap
dataset_ids:
  - SMRU_SEALDATA_2019-2020
  - SMRU_SEALDATA_2020-2021
folder: harvest
log_level: INFO
EOF

# 2. Harvest only those datasets
python -m cde_harvester -f harvest_config.yaml

# 3. Load incrementally
python -m cde_db_loader --folder harvest --incremental
```

### Example 2: Full Reload from All ERDDAP Servers

```bash
# 1. Edit harvest_config.yaml to harvest everything
cat > harvest_config.yaml <<EOF
erddap_urls:
  - https://cioos-atlantic-ckan.cioosatlantic.ca/erddap
  - https://catalogue.hakai.org/erddap
  - https://www.smartatlantic.ca/erddap
# No dataset_ids filter = harvest all
folder: harvest
log_level: INFO
EOF

# 2. Harvest all datasets
python -m cde_harvester -f harvest_config.yaml

# 3. Full reload
python -m cde_db_loader --folder harvest
```

### Example 3: Daily Incremental Updates

For a cron job that updates recently changed datasets:

```bash
#!/bin/bash
# daily_update.sh

# Harvest all datasets (or use change detection in future)
python -m cde_harvester -f harvest_config.yaml

# Load incrementally to avoid downtime
python -m cde_db_loader --folder harvest --incremental
```

## Database Performance

### Indexes

The profiles table has been optimized with indexes for incremental updates:

- `(erddap_url, dataset_id)`: Fast deletion of old profiles when updating a dataset
- `(erddap_url, dataset_id, timeseries_id, profile_id)`: Fast lookups for specific profiles
- Existing GIST indexes on `geom`, `hex_zoom_0`, `hex_zoom_1` for spatial queries

### Performance Comparison

| Operation | Full Reload | Incremental (10 datasets) | Incremental (100 datasets) |
|-----------|-------------|---------------------------|----------------------------|
| Typical Time | 5-30 min | 30 sec - 2 min | 3-10 min |
| Database Locks | Full table locks | Row-level locks | Row-level locks |
| Downtime | Yes (during reload) | Minimal | Minimal |
| Constraints | Drop & recreate | Keep existing | Keep existing |

## Command-Line Options

```
python -m cde_db_loader --help

Options:
  --folder TEXT        Folder with CSV output files from harvesting (default: "harvest")
  --incremental        Use UPSERT instead of deleting all data - only update/insert changed datasets
  --help              Show this message and exit
```

## Input Files

The loader expects these CSV files in the specified folder:

- `datasets.csv`: Dataset metadata (one row per dataset)
- `profiles.csv`: Profile/timeseries data (one row per profile)
- `skipped.csv`: Datasets that were skipped during harvesting

## Environment Variables

Required in `.env` file:

```bash
DB_HOST_EXTERNAL=localhost
DB_PORT=5432
DB_NAME=cde
DB_USER=postgres
DB_PASSWORD=yourpassword
SENTRY_DSN=optional_sentry_url
ENVIRONMENT=development
INCREMENTAL_MODE=false  # Set to "true" to enable incremental mode by default
```

**Note**: The `INCREMENTAL_MODE` environment variable can be used to set the default mode. The `--incremental` command-line flag overrides this setting.

## Technical Details

### Architecture

The db-loader has been refactored for better maintainability and performance:

**Python Code** ([`__main__.py`](cde_db_loader/__main__.py)):
- Helper functions for data preparation (`prepare_profiles_dataframe`, `ensure_organization_pks`)
- Constants for array column types (`DATASET_ARRAY_DTYPES`)
- Minimal SQL - most logic delegated to database functions
- **Incremental mode**: Just 25 lines (was 140 lines before refactoring)

**SQL Functions** ([`database/`](../database/)):
- [`5_profile_process.sql`](../database/5_profile_process.sql):
  - `process_profile_geometry_and_links(table)` - Shared geometry/linking logic
  - `process_temp_profiles()` - Process temp tables during incremental mode
  - `profile_process()` - Full reload processing
- [`9_incremental_upsert.sql`](../database/9_incremental_upsert.sql):
  - `create_temp_tables()` - Create staging tables
  - `upsert_datasets_from_temp()` - UPSERT datasets
  - `replace_profiles_from_temp()` - Replace profiles
  - `upsert_skipped_datasets_from_temp()` - UPSERT skipped datasets
  - `process_incremental_update()` - **Main orchestrator** (single entry point)

## Troubleshooting

### Incremental mode fails with constraint errors

If incremental mode fails, you may need to run a full reload first to ensure proper schema state:

```bash
python -m cde_db_loader --folder harvest  # Without --incremental
```

### Datasets not updating

Make sure the `dataset_id` and `erddap_url` in your CSV exactly match the database. The UPSERT uses these as the unique key.

### Missing profiles after incremental load

Incremental mode deletes ALL old profiles for updated datasets before inserting new ones. This is expected behavior to ensure consistency.
